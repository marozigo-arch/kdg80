# E2E-план для системы регистрации через Gherkin и behave

Документ фиксирует, как должна тестироваться будущая система регистрации до и после реализации, и закрепляет, что сценарии описываются в Gherkin и исполняются через `behave`.

Связанный канонический документ:

- [docs/registration-system-requirements.md](/workspaces/kdg80/docs/registration-system-requirements.md)
- [docs/behave/registration-system.feature](/workspaces/kdg80/docs/behave/registration-system.feature)

## 1. Цели тестового контура

- Проверить пользовательский путь регистрации через браузер на каждом событии. — Статус: `Зафиксировано`
- Проверить все обязательные валидации: корректные, некорректные и мусорные данные. — Статус: `Зафиксировано`
- Проверить конкурентную запись на последние места. — Статус: `Зафиксировано`
- Проверить генерацию билета, PDF и Telegram-уведомления суперадмину. — Статус: `Зафиксировано`
- Проверить поведение системы при временной недоступности Telegram. — Статус: `Зафиксировано`

## 2. Принципиальная среда для E2E

- Полный E2E для Telegram webhook должен гоняться не только локально, а на публичном Fly staging URL с HTTPS. — Статус: `Зафиксировано`
- Локально допускаются только частичные тесты формы и API с моками Telegram. — Статус: `Зафиксировано`
- Для E2E желательно отдельное staging-приложение и отдельный test bot, чтобы не засорять production-чат. — Статус: `Зафиксировано`
- Публичная форма в E2E должна отправлять запрос напрямую на Fly staging API, а не в обход через локальные моки формы. — Статус: `Зафиксировано`
- Для day-1 MVP production-like E2E может использовать штатный `.fly.dev` HTTPS URL приложения, если custom domain на Fly ещё не подключён. — Статус: `Зафиксировано`

### Production-first hygiene

- Для day-1 допустим production-first smoke/E2E, но только по выделенным test events, не участвующим в реальной продаже мест. — Статус: `Зафиксировано`
- Полный destructive и конкурентный suite не должен бить по реальным production events. — Статус: `Зафиксировано`
- Каждый автоматический прогон должен иметь уникальный `test_run_id`. — Статус: `Зафиксировано`
- После завершения production-like прогона тестовые данные должны очищаться отдельной superadmin-only командой `/cleanup_test_run <run_id>` или эквивалентным maintenance-сценарием. — Статус: `Зафиксировано`

## 3. Инструменты

- BDD runner: `behave`. — Статус: `Зафиксировано`
- Browser E2E: Playwright. — Статус: `Зафиксировано`
- Telegram assertions: Telethon с сессией из `TELEGRAM_SESSION_S22`. — Статус: `Зафиксировано`
- Дополнительные сервисы очередей для тестов не нужны. — Статус: `Зафиксировано`

### Формат сценариев

- Сценарии являются source of truth в Gherkin-файлах для `behave`. — Статус: `Зафиксировано`
- Playwright и Telethon вызываются из step definitions, а не через отдельный неформализованный набор скриптов. — Статус: `Зафиксировано`
- Каждый Gherkin-сценарий должен читаться как воспроизводимый тестовый скрипт: кто именно что делает на сайте, в Telegram или в тестовом окружении. — Статус: `Зафиксировано`
- Основные акторы Gherkin-сценариев: `посетитель сайта`, `суперадмин в Telegram`, `оператор в Telegram`, `тестовый scheduler`, `клиент аварийного export endpoint`. — Статус: `Зафиксировано`
- Архитектурные и внутренние утверждения вида `backend generated ...`, `database stores ...`, `outbox contains ...` не должны быть основным текстом пользовательского Gherkin-сценария. — Статус: `Зафиксировано`
- Такие внутренние проверки допустимы как скрытые assertions внутри step definitions или как отдельные integration checks, привязанные к пользовательскому сценарию. — Статус: `Зафиксировано`

### Замечание по секретам

- `TELEGRAM_SESSION_S22` должен храниться только локально в `.env`-файле, не попадающем в git. — Статус: `Зафиксировано`
- В текущем состоянии репозитория `docs/.env` не отслеживается git и покрывается правилом `.env` из корневого `.gitignore`. — Статус: `Зафиксировано`

## 4. Минимальный тестовый датасет

Нужны как минимум такие события:

- `Тест Научная библиотека — валидация формы` — открытое тестовое событие для негативных проверок формы. — Статус: `Зафиксировано`
- `Тест Последнее место` — событие, где остался `1` билет. — Статус: `Зафиксировано`
- `Тест Фридландские ворота — мест нет` — событие без мест. — Статус: `Зафиксировано`
- `Тест Архив — прошедшее событие` — событие в прошлом. — Статус: `Зафиксировано`
- Не меньше одного открытого события на каждую площадку из матрицы залов. — Статус: `Зафиксировано`

## 5. Базовые E2E-сценарии

### A. Bootstrap Telegram-бота

1. Первый тестовый пользователь отправляет `/start`.
2. Проверяем, что он становится суперадмином.
3. Проверяем, что после `/start` бот показывает кнопочную навигацию.
4. Проверяем, что команда `/help` выводит список доступных команд.
5. Второй тестовый пользователь отправляет `/start`.
6. Проверяем, что второй пользователь не становится админом и получает отказ/нейтральный ответ.

Ожидаемый результат:

- в таблице `telegram_admin` только одна запись;
- суперправа закреплены за первым пользователем.

### B. Happy path регистрации на каждом открытом событии

Для каждого открытого события в текущем каталоге:

1. Playwright открывает `/programma/`.
2. Прокручивает к нужному месяцу программы.
3. Находит конкретную карточку события по названию и площадке.
4. Нажимает CTA `Регистрация`.
5. Проверяет, что открылась форма регистрации с summary выбранного события.
6. Заполняет валидные `ФИО`, `email`, `телефон`.
7. Отправляет форму и убеждается, что submit уходит на Fly registration API по HTTPS.
8. Проверяет success-state и открытие `/ticket/<public_hash>` на основном домене сайта.
9. Проверяет наличие короткого `ticket ID`.
10. Проверяет кнопку `Скачать PDF`.
11. Проверяет блок `Добавьте событие в календарь, чтобы не забыть`.
12. Проверяет доступность `Google Calendar`, `iPhone / Apple Calendar`, `Android / ICS`, `Скачать ICS`.
13. Telethon ждёт уведомление в чате суперадмина.

Ожидаемый результат:

- регистрация создана;
- билет создан;
- публичный билет доступен по длинному `public_hash`;
- пользователю показан `short_ticket_id`;
- на билете есть полный адрес площадки;
- на билете есть блок `Добавьте событие в календарь, чтобы не забыть`;
- на билете доступны platform-friendly варианты календаря, а не одна универсальная непонятная ссылка;
- в сообщении бота есть ФИО, название события, дата/время и остаток мест.

Для читаемости behave-сценариев и Playwright-step definitions используются конкретные публичные события из мастер-файла, например:

- `Виштынецкая возвышенность: как осваивали с 1945 года, современность и перспективы` — `/programma/` -> `Апрель 2026` -> Научная библиотека
- `Природа чемодана` — `/programma/` -> `Апрель 2026` -> ИЦАЭ, КГТУ
- `Советское монументальное искусство на территории Калининградской области` — `/programma/` -> `Апрель 2026` -> Музей «Фридландские ворота»
- `Разговоры о море` — `/programma/` -> `Апрель 2026` -> Музей Мирового океана
- `Памятники искусства и истории в ландшафте Калининградского зоопарка` — `/programma/` -> `Июнь 2026` -> Филиал Государственной Третьяковской галереи в Калининграде

### C. Валидация корректных и некорректных данных

Отдельные Playwright-сценарии:

1. Пустое ФИО.
2. ФИО из пробелов.
3. Email без `@`.
4. Email с опечаткой в домене.
5. Email из disposable-domain списка.
6. Телефон не на `+7`.
7. Телефон короче и длиннее допустимого.
8. Поля с мусором: HTML, JS, SQL-like строки, emoji, очень длинные строки.
9. Дубль email.
10. Дубль телефона.
11. Submit без галочки согласия на обработку ПДн.

Ожидаемый результат:

- пользователь видит понятную inline-ошибку;
- сервер не создаёт регистрацию;
- в логах и UI нет XSS/unsafe rendering.

### C2. Submit feedback и защита от “нажал и ничего не происходит”

1. Посетитель открывает форму регистрации на конкретном событии.
2. Заполняет валидные данные.
3. Нажимает submit.
4. Проверяется, что кнопка сразу меняет состояние на `Регистрируем…` и становится disabled.
5. При искусственно замедленном ответе сервера проверяется промежуточный текст о резервировании места.
6. Проверяется, что повторный клик не создаёт второй submit.

Ожидаемый результат:

- пользователь видит явный loading-state;
- форма не зависает молча;
- повторная отправка из-за нетерпеливого double-click не создаёт дубль.

### D. Запрет регистрации на прошлое событие

1. Playwright открывает `Тест Архив — прошедшее событие`.
2. Проверяет отсутствие активной регистрации или серверный отказ при прямом POST.

Ожидаемый результат:

- UI показывает, что событие прошло;
- прямой POST тоже отклоняется на сервере.

### E. Продажа последнего места

1. `Тест Последнее место` имеет ровно `1` свободное место.
2. Два или более параллельных клиента одновременно отправляют форму.

Ожидаемый результат:

- ровно одна регистрация успешна;
- остальные получают `мест нет` или мягкий retry-ответ, если конфликт был до финальной проверки;
- отрицательных остатков нет.

### F. Массовая конкурентная попытка

1. Для события с лимитом запускается пачка параллельных POST-запросов.
2. Количество попыток превышает число свободных мест.

Ожидаемый результат:

- итоговое число регистраций строго равно доступной квоте;
- `seats_taken` не превышает `capacity`;
- база не уходит в долгий lock.

## 6. Telegram-specific сценарии

### Уведомление о регистрации

1. После браузерной регистрации Telethon проверяет входящее сообщение суперадмину.
2. Проверяются поля:
   - ФИО
   - событие
   - дата и время
   - остаток мест

### Поиск по ФИО

1. Через браузер создаётся регистрация.
2. Через Telethon в бот отправляется `/find <ФИО>`.
3. Проверяется, что бот находит запись.

### Отчёт по событию и XLSX

1. Через бот запрашивается список событий.
2. Выбирается событие.
3. Запрашивается экспорт.
4. В интерфейсе бота проверяется, что телефон и email показаны маскированно.
5. Проверяется XLSX:
   - название события
   - ФИО
   - почта без маски
   - телефон без маски
   - ссылка на приглашение

### Открытие и закрытие регистрации через Telegram

1. Суперадмин открывает раздел `События` кнопкой.
2. Выбирает конкретное событие кнопкой из списка.
3. Нажимает кнопку `Открыть регистрацию` на event-card.
4. Проверяется, что событие стало доступно для внутренней регистрации и карточка обновилась.
5. Затем суперадмин нажимает кнопку `Закрыть регистрацию`.
6. Проверяется, что регистрация снова недоступна и прямой POST отклоняется.
7. Оператор без прав суперадмина пытается выполнить тот же сценарий.
8. Проверяется, что бот отклоняет действие и не меняет состояние события.

### Help и кнопочная навигация

1. После `/start` бот показывает клавиатуру с основными действиями.
2. По кнопке `Помощь` открывается тот же список команд, что и по `/help`.
3. Для суперадмина видны кнопки открытия/закрытия регистрации.
4. Для суперадмина видна кнопка управления операторами.
5. Для оператора кнопки открытия/закрытия регистрации не показываются.
6. Из раздела `События` суперадмин может дойти до конкретного события и сменить его registration-state кнопкой без ручного ввода команды.

### Daily export и SQL dump

1. Тест принудительно запускает daily job или ждёт тестовый schedule.
2. Проверяет, что суперадмин получил сводный XLSX.
3. Проверяет, что суперадмин получил SQL dump/backup.
4. Проверяет, что задача срабатывает в low-load окне `Europe/Kaliningrad`.

### Self-heal после рестарта приложения

1. В тестовом окружении приложение принудительно перезапускается.
2. Проверяется, что после старта бот снова отвечает на `/help`.
3. Проверяется, что webhook остаётся рабочим без ручной переустановки.
4. Проверяется, что outbox worker и daily scheduler снова активны.

## 7. Сценарии недоступности Telegram

### Кратковременный outage Telegram API

1. В тестовой среде блокируется исходящий доступ приложения к Telegram API.
2. Пользователь проходит регистрацию через браузер.
3. Проверяется, что регистрация и билет созданы успешно.
4. Проверяется, что запись появилась в `telegram_outbox`.
5. Доступ к Telegram API восстанавливается.
6. Проверяется, что накопленное уведомление дошло позже.

Ожидаемый результат:

- регистрация не теряется;
- outbox не застревает навсегда;
- уведомление становится eventual, а не blocking.

### Недоступность webhook endpoint

1. Telegram пытается достучаться до staging webhook.
2. Проверяется `getWebhookInfo` и служебные логи приложения.

Ожидаемый результат:

- команда мониторинга видит `pending_update_count` и последние ошибки;
- после восстановления endpoint бот продолжает работу без переустановки всей системы.

### Аварийный export endpoint

1. Telegram недоступен длительное время.
2. Суперадмин обращается к защищённому read-only export endpoint с секретом в `Authorization` header.
3. Проверяется, что endpoint не даёт менять состояние регистрации.
4. Проверяется, что endpoint не доступен без корректного секрета.

### Очистка тестового прогона

1. Automated run с `test_run_id` создаёт тестовые регистрации.
2. Через Telethon суперадмин запускает `/cleanup_test_run <run_id>`.
3. Проверяется, что тестовые регистрации и связанные ticket artifacts удалены.
4. Проверяется, что `seats_taken` после очистки пересчитан корректно.

## 8. Проверка билета и PDF

1. После успешной регистрации Playwright открывает `/ticket/<public_hash>`.
2. Проверяет, что страница доступна и содержит `short_ticket_id`.
3. Проверяет, что неиндексируемые страницы отдают корректный `noindex`.
4. Проверяет, что на странице есть полный адрес площадки.
5. Проверяет, что на странице есть блок `Добавьте событие в календарь, чтобы не забыть`.
6. Проверяет, что доступны `Google Calendar`, `iPhone / Apple Calendar`, `Android / ICS`, `Скачать ICS`.
7. Проверяет, что `Google Calendar` открывает корректно заполненный calendar URL, а `Apple/ICS` ведут на статический `.ics`-файл билета.
8. Проверяет, что по кнопке скачивается PDF.
9. Проверяет, что в PDF есть `short_ticket_id`, ссылка на `/ticket/<public_hash>`, полный адрес и пометка `Печать билета не требуется`.
10. Проверяет, что пользователь не видит на ticket-page self-service действия отмены регистрации.

## 9. Что именно должно автоматизироваться через Playwright

- Навигация к форме регистрации.
- Работа формы на desktop viewport.
- Работа формы на mobile viewport.
- Проверка input mask и UX телефона.
- Проверка inline validation.
- Проверка happy path.
- Проверка sold-out и past-event состояний.

## 10. Какие внутренние проверки всё равно нужны в step definitions

- После успешной регистрации step definitions дополнительно проверяют, что ticket HTML, PDF и `.ics` реально доступны по ожидаемым URL. — Статус: `Зафиксировано`
- После успешной регистрации внутренние проверки убеждаются, что в БД нет plaintext `ФИО`, `email` и `телефона`, а есть только шифротекст и blind indexes. — Статус: `Зафиксировано`
- Во время Telegram outage внутренние проверки убеждаются, что уведомление поставлено в `telegram_outbox` и позже доставлено. — Статус: `Зафиксировано`
- В конкурентных сценариях внутренние проверки убеждаются, что `seats_taken` не превысил `capacity`. — Статус: `Зафиксировано`
- В сценариях duplicate checks внутренние проверки убеждаются, что в БД не появилась вторая регистрация для того же `event_id`. — Статус: `Зафиксировано`
- В production-like тестовых прогонах внутренние проверки убеждаются, что cleanup по `test_run_id` действительно удалил тестовые строки и вернул счётчики мест. — Статус: `Зафиксировано`

## 11. Критерии приёмки E2E-контура

- Все happy-path сценарии проходят на каждом открытом событии. — Статус: `Зафиксировано`
- Ни один негативный тест не создаёт “битую” регистрацию. — Статус: `Зафиксировано`
- При конкурентной записи система никогда не отдаёт больше билетов, чем есть мест. — Статус: `Зафиксировано`
- При Telegram outage регистрация продолжает работать. — Статус: `Зафиксировано`
- После восстановления Telegram backlog уведомлений догоняется. — Статус: `Зафиксировано`
- Event-level экспорт доступен из бота, а общий экспорт и SQL dump доступны суперадмину. — Статус: `Зафиксировано`
- Ticket page даёт понятные calendar actions для Google и `.ics`-сценариев. — Статус: `Зафиксировано`

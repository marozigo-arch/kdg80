# Система регистрации фестиваля — требования и MVP-схема

Этот документ фиксирует минимальную, быстро реализуемую схему кастомной регистрации для фестивального сайта без перехода к реализации в текущем шаге.

Связанные документы:

- [docs/festival-landing-requirements.md](/workspaces/kdg80/docs/festival-landing-requirements.md)
- [docs/po-interesam-requirements.md](/workspaces/kdg80/docs/po-interesam-requirements.md)

## Статусы

- `Зафиксировано` — требование закреплено как источник истины для MVP.
- `Не выполнено` — требование принято, но в коде ещё не реализовано.
- `Не подтверждено пользователем` — внутренне может быть проверено позже, но пользователь ещё не подтвердил вручную.
- `В доработке` — требование или бизнес-правило ещё требует уточнения.

## Состояние пакета на сейчас

- Технический scope day-1 MVP зафиксирован. — Статус: `Зафиксировано`
- Документация готова к пользовательской проверке перед реализацией. — Статус: `Не подтверждено пользователем`

- Отдельного внешнего подтверждения всё ещё требует только юридический текст согласия на обработку ПДн. — Статус: `В доработке`

## 2. Цель MVP

- MVP должен быть реализуем за один рабочий день без многонодовой и многосервисной схемы. — Статус: `Зафиксировано`
- MVP должен работать на Fly.io и использовать только внутреннюю SQL-базу приложения. — Статус: `Зафиксировано`
- MVP не должен зависеть от доступности Telegram для факта успешной регистрации пользователя. — Статус: `Зафиксировано`
- Telegram остаётся операторским интерфейсом мониторинга и администрирования. Пользовательская регистрация на события идёт не через Telegram, а через форму сайта и прямой API Fly.io. — Статус: `Зафиксировано`
- В MVP сознательно допускается компромисс: одна машина и один volume ради скорости запуска, с пониманием, что это не high-availability конфигурация. — Статус: `Зафиксировано`

## 3. Архитектурный канон MVP

- Публичный сайт и опубликованные ticket-страницы живут на статическом хостинге проекта, а Fly.io приложение обслуживает API регистрации, генерацию ticket artifacts и Telegram webhook. — Статус: `Зафиксировано`
- На Fly.io используется `single region + single always-on machine + attached volume`. — Статус: `Зафиксировано`
- Приложение не должно scale-to-zero, иначе webhook и daily jobs станут ненадёжными. — Статус: `Зафиксировано`
- База данных MVP — SQLite на примонтированном Fly Volume. Других транзакционных хранилищ в MVP нет. — Статус: `Зафиксировано`
- Публичный сайт может оставаться статическим на текущем хостинге; перевод всего сайта в server/hybrid-режим для MVP не требуется, потому что регистрационный backend живёт отдельно на Fly.io. — Статус: `Зафиксировано`
- Redis, RabbitMQ, отдельная очередь, LiteFS, отдельный admin frontend и multi-region репликация в MVP не включаются. — Статус: `Зафиксировано`
- Временная зона всей серверной логики регистрации и daily export фиксируется как `Europe/Kaliningrad`. — Статус: `Зафиксировано`

### Зафиксированный runtime stack MVP

- Backend runtime: `Node.js 22`. — Статус: `Зафиксировано`
- HTTP framework: `Fastify`. — Статус: `Зафиксировано`
- SQLite driver: `better-sqlite3`. — Статус: `Зафиксировано`
- Миграции БД: простые hand-written SQL migrations, применяемые самим приложением на старте. — Статус: `Зафиксировано`
- Telegram bot framework: `grammY`. — Статус: `Зафиксировано`
- XLSX export: `ExcelJS`, только вне горячего пути регистрации. — Статус: `Зафиксировано`
- PDF generation: `PDFKit` без headless browser. — Статус: `Зафиксировано`
- HTML ticket page собирается из лёгкого серверного шаблона без отдельного SSR-фреймворка админки. — Статус: `Зафиксировано`

### Минимальная логическая схема

```text
Browser form on public site
  -> HTTPS POST to Fly API /api/v1/register
    -> validate + normalize
    -> SQLite transaction
      -> reserve seat
      -> create registration
      -> create ticket
      -> enqueue tg_outbox job
    -> generate and upload ticket HTML/PDF/ICS outside transaction
    -> success page /ticket/<public_hash> on main site domain

Telegram
  -> POST /api/telegram/webhook
    -> command handling
    -> reports / exports / search

Background loop inside same app
  -> send tg_outbox notifications
  -> run daily exports
  -> create DB dump
```

## 4. Требования к Fly.io-деплою

- Деплой делается как одно приложение с одной основной process group `app`. — Статус: `Зафиксировано`
- Fly Volume монтируется в каталог данных приложения, например `/data`, и только там хранится SQLite-файл и производные backup/export-артефакты. — Статус: `Зафиксировано`
- Машина должна быть always-on, минимум `1` running machine. — Статус: `Зафиксировано`
- Для single-volume деплоя нужно заранее принять короткий downtime на обновлении вместо попытки имитировать zero-downtime с конфликтом volume attachment. — Статус: `Зафиксировано`
- Публичный домен должен обслуживаться только по HTTPS. — Статус: `Зафиксировано`
- В приложении должен быть отдельный health endpoint для Fly health checks. — Статус: `Не выполнено`
- Для MVP допустим production API на штатном `https://<app>.fly.dev`, без обязательного custom domain на Fly.io. — Статус: `Зафиксировано`
- На Fly.io должна быть выставлена restart policy, обеспечивающая автоматический перезапуск приложения после падения или ошибочного рестарта. — Статус: `Зафиксировано`
- Для webhook и scheduler-контуров `auto_stop` должен быть выключен, чтобы приложение не засыпало между запросами. — Статус: `Зафиксировано`

## 5. База данных и защита от lock/овербукинга

- При старте соединения должны быть включены как минимум `PRAGMA foreign_keys = ON`, `PRAGMA journal_mode = WAL`, `PRAGMA busy_timeout`, а запись должна выполняться короткими транзакциями. — Статус: `Зафиксировано`
- Истина о свободных местах определяется только базой данных, а не расчётом в UI. — Статус: `Зафиксировано`
- Побеждает тот пользователь, чья транзакция записи первой успешно закоммитилась. Это и есть правило разрешения конкурентной гонки. — Статус: `Зафиксировано`
- При исчерпании мест пользователь должен получить финальный ответ `мест нет`, а не “успех с последующим откатом вручную”. — Статус: `Зафиксировано`
- При кратковременном конфликте записи пользователь может получить мягкий ответ `попробуйте ещё раз через 10–15 секунд`, если lock не удалось взять за разумный timeout. — Статус: `Зафиксировано`
- Никакие внешние HTTP-вызовы, включая Telegram API, не выполняются внутри транзакции регистрации. — Статус: `Зафиксировано`

### Рекомендуемая транзакционная модель

- У события хранится `capacity` и счётчик `seats_taken`. — Статус: `Зафиксировано`
- Сервер валидирует, что событие ещё не началось и регистрация на него разрешена. — Статус: `Зафиксировано`
- Внутри короткой транзакции выполняется атомарный `UPDATE events ... WHERE seats_taken < capacity AND starts_at > now`. — Статус: `Зафиксировано`
- Если `UPDATE` не изменил ни одной строки, сервер возвращает либо `мест нет`, либо `событие прошло/закрыто`, в зависимости от причины. — Статус: `Зафиксировано`
- После успешного резервирования места создаются `registration`, `ticket` и запись в `tg_outbox`; затем транзакция коммитится. — Статус: `Зафиксировано`
- Если вставка регистрации падает по unique constraint, транзакция целиком откатывается и место не теряется. — Статус: `Зафиксировано`

### Уникальность дублей

- Дубли по `email` и `телефону` запрещаются только в рамках одного и того же события. — Статус: `Зафиксировано`
- Один и тот же человек может зарегистрироваться на разные события фестиваля, если нет конфликта по лимиту мест и времени. — Статус: `Зафиксировано`
- Уникальные ограничения должны строиться на `(event_id, email_fingerprint)` и `(event_id, phone_fingerprint)`, а не глобально на весь фестиваль. — Статус: `Зафиксировано`
- Если у события регистрация открыта, она всегда идёт через внутреннюю систему регистрации. Внешние registration links для открытых регистраций не используются. — Статус: `Зафиксировано`

## 7. Telegram-бот как операторский интерфейс

- Первый пользователь, который первым отправил боту `/start`, становится суперадмином. — Статус: `Зафиксировано`
- После назначения суперадмина бот больше не регистрирует новых администраторов автоматически. Новые операторы назначаются только через действие суперадмина. — Статус: `Зафиксировано`
- Роль `operator` должна быть заложена в MVP. — Статус: `Зафиксировано`
- Для каждой успешной регистрации бот должен отправлять суперадмину уведомление с ФИО, событием, датой/временем и остатком мест. — Статус: `Зафиксировано`
- Бот должен уметь показать список событий и кнопку вызова отчёта по каждому событию. — Статус: `Зафиксировано`
- Бот должен уметь искать регистрация по ФИО. — Статус: `Зафиксировано`
- Бот должен уметь отдать XLSX по событию с колонками `Название события`, `ФИО`, `Почта`, `Телефон`, `Ссылка на приглашение`. — Статус: `Зафиксировано`
- Бот должен раз в сутки экспортировать общую таблицу всех билетов и дамп SQL-базы. — Статус: `Зафиксировано`
- Интерфейс бота должен быть максимально кнопочным: сценарии просмотра событий, открытия/закрытия регистрации, вызова отчётов и выгрузок должны быть доступны через кнопки, а не только через ручной ввод команд. — Статус: `Зафиксировано`
- Telegram подключается только через webhook. — Статус: `Зафиксировано`
- После старта или рестарта приложения бот должен сам проверить актуальность webhook и при необходимости перевыставить его без ручного вмешательства. — Статус: `Зафиксировано`
- После рестарта приложения бот должен продолжать работу с уже существующим суперадмином и операторами, не требуя повторного bootstrap через `/start`. — Статус: `Зафиксировано`

### Минимальный набор команд и действий

- `/start` — claim суперадмина или сообщение, что админ уже назначен; после входа показывается кнопочная навигация по доступным действиям. — Статус: `Зафиксировано`
- `/help` — список всех доступных команд и кратких пояснений по ним. — Статус: `Зафиксировано`
- `/operators` — просмотр операторов и выдача/отзыв роли суперадмином. — Статус: `Зафиксировано`
- `/events` — список событий с inline-кнопками `Отчёт`, `XLSX`, `Остаток мест`. — Статус: `Зафиксировано`
- `/registration_open <event>` — открыть регистрацию на событие; право только у суперадмина. — Статус: `Зафиксировано`
- `/registration_close <event>` — закрыть регистрацию на событие; право только у суперадмина. — Статус: `Зафиксировано`
- `/find <ФИО>` — поиск по имени. — Статус: `Зафиксировано`
- `/export_all` — сводный XLSX по всем событиям; право только у суперадмина. — Статус: `Зафиксировано`
- `/backup_sqlite` — выгрузка дампа базы; право только у суперадмина. — Статус: `Зафиксировано`
- `/cleanup_test_run <run_id>` — очистка тестовых регистраций конкретного автопрогона; право только у суперадмина. — Статус: `Зафиксировано`
- Оператору доступны мониторинг, поиск и event-level отчёты/XLSX в Telegram, но не глобальный export, не SQL dump и не изменение состояния регистрации. — Статус: `Зафиксировано`
- Командный ввод остаётся резервным интерфейсом, но не должен быть основным способом повседневной работы суперадмина и оператора. — Статус: `Зафиксировано`

### Навигация после `/start`

- После `/start` бот должен сразу показывать кнопки основных действий, чтобы администратору не приходилось помнить команды наизусть. — Статус: `Зафиксировано`
- Минимальный набор кнопок для суперадмина: `События`, `Поиск`, `Экспорт`, `Открыть регистрацию`, `Закрыть регистрацию`, `Операторы`, `Помощь`. — Статус: `Зафиксировано`
- Минимальный набор кнопок для оператора: `События`, `Поиск`, `Экспорт`, `Помощь`. — Статус: `Зафиксировано`
- При входе в раздел `События` бот должен показывать paginated list событий кнопками, без необходимости вводить slug вручную. — Статус: `Зафиксировано`
- Для выбранного события бот должен открывать отдельную event-card с минимумом кнопок: `Открыть регистрацию` или `Закрыть регистрацию`, `Отчёт`, `XLSX`, `Остаток мест`, `Назад`. — Статус: `Зафиксировано`
- Кнопка `Открыть регистрацию` показывается суперадмину только если событие допускает открытие, а `Закрыть регистрацию` — только если регистрация сейчас открыта. — Статус: `Зафиксировано`
- После нажатия кнопки смены статуса бот должен сразу обновлять event-card и показывать новый публичный registration-state события. — Статус: `Зафиксировано`
- Callback-обработчики кнопок в Telegram должны быть идемпотентны, чтобы двойное нажатие не приводило к лишним ошибкам и рассинхронизации интерфейса. — Статус: `Зафиксировано`

## 8. Недоступность Telegram и не-избыточная деградация

- Регистрация пользователя не должна ждать синхронного ответа от Telegram API. — Статус: `Зафиксировано`
- Все Telegram-действия после регистрации должны сначала попадать в `telegram_outbox`. — Статус: `Зафиксировано`
- Фоновый воркер внутри того же приложения должен забирать `telegram_outbox` пачками и ретраить доставку с backoff. — Статус: `Зафиксировано`
- Если Telegram недоступен, регистрации продолжают создаваться, билеты продолжают генерироваться, а уведомления и daily-экспорты догоняются после восстановления связи. — Статус: `Зафиксировано`
- Для MVP не делается отдельная отказоустойчивая шина событий только ради Telegram. — Статус: `Зафиксировано`
- Допустим минимальный аварийный fallback без полноценной админки: защищённый read-only export endpoint для суперадмина на случай долгой недоступности Telegram. — Статус: `Зафиксировано`
- Аварийный export endpoint должен быть HTTPS-only, read-only, отдельным от публичного registration API и защищён отдельным длинным secret token из Fly secrets; этот токен не используется в браузерной форме регистрации. — Статус: `Зафиксировано`
- После рестарта приложения outbox worker и daily scheduler должны автоматически подниматься заново и продолжать работу без ручного запуска. — Статус: `Зафиксировано`

## 9. Публичная форма регистрации и UX

- Обязательные поля: `ФИО`, `email`, `телефон`. — Статус: `Зафиксировано`
- На форме явно показывается, что каждый участник регистрируется сам за себя. — Статус: `Зафиксировано`
- В footer и рядом с формой должна быть подсказка: `По вопросам, связанным с билетами, пишите на bilet@kdg80.ru`. — Статус: `Зафиксировано`
- Поля должны валидироваться и на клиенте, и на сервере. Сервер — источник истины. — Статус: `Зафиксировано`
- Ошибки показываются рядом с полем, а не только общим баннером сверху. — Статус: `Зафиксировано`
- На mobile форма должна использовать корректные типы ввода и клавиатуры. — Статус: `Зафиксировано`
- В MVP не добавляется CAPTCHA по умолчанию, чтобы не бить по конверсии; базовая антиспам-защита строится на honeypot, rate limit, валидации и коротких server-side ограничениях. — Статус: `Зафиксировано`

### Где и как вызывается форма регистрации

- Основная точка входа в регистрацию — CTA `Регистрация` в карточке события на странице `/programma/`. — Статус: `Зафиксировано`
- Та же CTA `Регистрация` может быть показана на главной и на тематических страницах, если там переиспользуется карточка события с открытой регистрацией. — Статус: `Зафиксировано`
- Если для события включена отдельная страница `/sobytiya/<slug>/`, на ней используется та же самая форма регистрации и тот же submit-flow, что и в карточке программы. — Статус: `Зафиксировано`
- Клик по `Регистрация` не должен уводить пользователя на внешний сайт или отдельный сервис. — Статус: `Зафиксировано`
- На desktop форма открывается в лёгком modal dialog поверх текущей страницы программы, чтобы не терять контекст выбранного события. — Статус: `Зафиксировано`
- На mobile форма открывается как full-screen sheet / full-screen dialog, а не как маленький модальный слой с тесными полями. — Статус: `Зафиксировано`
- В верхней части формы обязательно показывается summary выбранного события: название, дата, время, площадка, зал и полный адрес. — Статус: `Зафиксировано`
- В форме рядом с полями должен быть виден сервисный контакт `bilet@kdg80.ru`. — Статус: `Зафиксировано`
- После клика по CTA пользователь должен получить видимый отклик интерфейса сразу, без состояния “нажал и ничего не произошло”. — Статус: `Зафиксировано`
- Если пользователь закрыл форму без успешной отправки, введённые данные должны сохраняться хотя бы в рамках текущей страницы/сессии, пока вкладка не была перезагружена. — Статус: `Зафиксировано`
- В форме должен быть обязательный unchecked checkbox согласия на обработку персональных данных; без него submit невозможен. — Статус: `Зафиксировано`
- Текст checkbox должен ссылаться как минимум на политику обработки/конфиденциальности и на отдельный документ с полным текстом согласия. — Статус: `Зафиксировано`
- Чекбокс согласия не может быть предзаполнен по умолчанию. — Статус: `Зафиксировано`
- На момент регистрации система должна сохранять версию согласия, hash текста согласия и timestamp акцепта, чтобы потом можно было доказать, с какой редакцией согласился пользователь. — Статус: `Зафиксировано`
- Рабочая редакция текста согласия централизуется в [docs/registration-personal-data-consent.md](/workspaces/kdg80/docs/registration-personal-data-consent.md). — Статус: `Зафиксировано`

### Поля и поведение

- `ФИО`: `autocomplete="name"`, нормализация пробелов, минимум два смысловых блока через пробел. — Статус: `Зафиксировано`
- Для `ФИО` каждый обязательный блок должен содержать не меньше `2` буквенных символов после нормализации. Форматы вида `И И`, `А Б`, `И.` и одно слово должны отклоняться. — Статус: `Зафиксировано`
- Для `ФИО` разрешаются буквы кириллицы и латиницы, пробел, дефис и апостроф; цифры, emoji, HTML-теги, управляющие символы и мусорные вставки не допускаются. — Статус: `Зафиксировано`
- Для `ФИО` ввод длиннее `120` символов отклоняется. — Статус: `Зафиксировано`
- `Email`: `type="email"`, `autocomplete="email"`, trimming пробелов, серверная валидация формата, длины и блокировка доменов одноразовой почты. — Статус: `Зафиксировано`
- Для `Email` допустимы plus-alias адреса, если домен не относится к disposable-почте. — Статус: `Зафиксировано`
- Для `Email` ввод длиннее `254` символов отклоняется. — Статус: `Зафиксировано`
- `Телефон`: ввод с фиксированным префиксом `+7`, маской вида `+7 (___) ___-__-__`, mobile-friendly numeric keypad, хранение в нормализованном формате `+7XXXXXXXXXX`. — Статус: `Зафиксировано`
- Для `Телефона` сервер принимает paste с пробелами, скобками и дефисами, но после нормализации обязан остаться ровно российский номер `+7` и `10` цифр после кода страны. — Статус: `Зафиксировано`
- Прямой ввод другой страны, недостаточного/избыточного числа цифр и пустых значений отклоняется и на клиенте, и на сервере. — Статус: `Зафиксировано`
- Отсутствие галочки согласия должно давать inline-ошибку у checkbox-блока, а не только общий баннер. — Статус: `Зафиксировано`
- После успешной регистрации пользователь попадает на страницу билета `/ticket/<public_hash>` и видит постоянную ссылку на билет и кнопку скачивания PDF. — Статус: `Зафиксировано`
- На сервере нельзя зарегистрироваться на событие, которое уже началось или завершилось. — Статус: `Зафиксировано`
- Пользователь не может самостоятельно отменить регистрацию через публичный сайт, ticket-page или форму. — Статус: `Зафиксировано`

### Статусная модель CTA, формы и submit-потока

- Состояния registration-CTA являются derived presentation layer поверх канонических event statuses из [docs/festival-landing-requirements.md](/workspaces/kdg80/docs/festival-landing-requirements.md), а не отдельным несогласованным enum. — Статус: `Зафиксировано`
- Для карточки события фиксируются по меньшей мере такие состояния registration-CTA: `registration_open`, `registration_soon`, `registration_closed`, `sold_out`, `past`. — Статус: `Зафиксировано`
- Для различения состояний `registration_soon` и `registration_closed` сервер не должен опираться только на одно boolean-поле `registration_enabled`; нужен явный server-side registration state. — Статус: `Зафиксировано`
- В состоянии `registration_open` пользователь видит активную кнопку `Регистрация`. — Статус: `Зафиксировано`
- В состоянии `registration_soon` вместо кнопки показывается неактивная плашка `Регистрация на мероприятие скоро откроется.` Это состояние используется для будущего события, где места ещё есть, но публичный приём заявок ещё не открыт. — Статус: `Зафиксировано`
- В состоянии `registration_closed` вместо кнопки показывается неактивная плашка `Регистрация на это событие сейчас закрыта.` Это состояние используется для будущего события, где приём заявок был вручную закрыт или приостановлен суперадмином. — Статус: `Зафиксировано`
- В состоянии `sold_out` вместо активной кнопки показывается плашка `Свободные места закончились.` — Статус: `Зафиксировано`
- В состоянии `past` вместо активной кнопки показывается плашка `Событие прошло`. — Статус: `Зафиксировано`
- Если у события ещё есть места, но `registration_public_state = soon`, интерфейс обязан показывать именно `Регистрация на мероприятие скоро откроется.`, а не `Регистрация закрыта.` — Статус: `Зафиксировано`
- Если у события ещё есть места, но `registration_public_state = closed`, интерфейс обязан показывать именно `Регистрация на это событие сейчас закрыта.`, а не `Регистрация скоро откроется.` — Статус: `Зафиксировано`
- Для формы фиксируются такие пользовательские состояния: `idle`, `editing`, `field_error`, `submitting`, `slow_request`, `success`, `duplicate`, `sold_out`, `retryable_lock`, `network_error`, `server_error`. — Статус: `Зафиксировано`
- После нажатия на submit кнопка должна сразу стать disabled и сменить текст на `Регистрируем…`. — Статус: `Зафиксировано`
- В состоянии `submitting` пользователь не может породить повторную отправку множественными кликами. — Статус: `Зафиксировано`
- Если сервер отвечает дольше примерно `4` секунд, форма должна показать вспомогательный текст `Проверяем данные и резервируем место. Это может занять несколько секунд.` — Статус: `Зафиксировано`
- При `network_error` и `server_error` введённые пользователем данные сохраняются в форме, кнопка возвращается в активное состояние, а пользователь получает понятное сообщение о дальнейших действиях. — Статус: `Зафиксировано`
- После `success` пользователь должен автоматически перейти на ticket-page без промежуточного пустого экрана. — Статус: `Зафиксировано`

## 10. Билет и идентификаторы

- Билет живёт как статичная по URL страница `/ticket/<public_hash>`, а `public_hash` должен быть длинным непредсказуемым токеном, устойчивым к bruteforce-перебору. — Статус: `Зафиксировано`
- Для MVP используется `public_hash` длиной `43` символа в `base64url`, полученный из `32` криптографически случайных байт. — Статус: `Зафиксировано`
- У билета есть отдельный короткий человеко-читаемый идентификатор `short_ticket_id` длиной `6` символов по шаблону `^[0-9][A-Z0-9]{4}[0-9]$`. — Статус: `Зафиксировано`
- `short_ticket_id` показывается на ticket-page и в PDF. — Статус: `Зафиксировано`
- `public_hash` и `short_ticket_id` должны иметь уникальные индексы в БД, а генерация обязана использовать retry при конфликте, чтобы коллизии не доходили до пользователя. — Статус: `Зафиксировано`
- На странице `/ticket/<public_hash>` всегда доступна функция `Скачать PDF`. — Статус: `Зафиксировано`
- В PDF должна быть ссылка обратно на ticket-page `/ticket/<public_hash>`. — Статус: `Зафиксировано`
- На билете и ticket-page должно быть явно сказано: `Печать билета не требуется`. — Статус: `Зафиксировано`
- На билете должно быть явно сказано, что рассадка свободная. — Статус: `Зафиксировано`
- На ticket-page и в PDF должен быть полный адрес площадки, чтобы человек мог понять, куда ехать. — Статус: `Зафиксировано`
- Ticket pages не должны индексироваться поисковиками и не должны попадать в sitemap, `llms.txt` или публичные AI-ready списки страниц. — Статус: `Зафиксировано`
- Ticket pages не должны раскрывать лишние персональные данные сверх необходимого минимума. — Статус: `Зафиксировано`
- На публичной ticket-page показываются `ФИО полностью`, `телефон маскированно`, `email частично маскированно`. — Статус: `Зафиксировано`
- В Telegram-интерфейсе данные посетителя показываются маскированно. — Статус: `Зафиксировано`
- В XLSX, SQL dump и других operator exports данные выгружаются без масок. — Статус: `Зафиксировано`
- Детальный визуальный и шаблонный канон билета централизуется в [docs/ticket-design-brief.md](/workspaces/kdg80/docs/ticket-design-brief.md). — Статус: `Зафиксировано`

## 11. Календарные действия на ticket-page

- На ticket-page должен быть заметный блок `Добавьте событие в календарь, чтобы не забыть`. — Статус: `Зафиксировано`
- Пользователю не нужно догадываться, какой формат ему подходит; на странице должны быть явные варианты под разные устройства и сценарии. — Статус: `Зафиксировано`
- Минимальный набор действий: `Google Calendar`, `iPhone / Apple Calendar`, `Android / ICS`, `Скачать ICS`. — Статус: `Зафиксировано`
- Календарные действия должны использовать те же event data, что и билет: название, дата, время, площадка, полный адрес и ссылка на ticket-page. — Статус: `Зафиксировано`
- Календарный блок должен быть доступен и на HTML-странице билета, и в PDF в виде понятной ссылки на страницу билета. — Статус: `Зафиксировано`
- Для MVP календарный контур должен быть максимально лёгким: `Google Calendar` ведёт на template URL, а `iPhone / Apple Calendar`, `Android / ICS` и `Скачать ICS` опираются на один и тот же статически сгенерированный `.ics`-файл билета. — Статус: `Зафиксировано`
- Календарный блок должен явно мотивировать пользователя не откладывать действие, например через формулировку `Добавьте событие в календарь сейчас, чтобы не забыть`. — Статус: `Зафиксировано`

## 12. Генерация HTML/PDF и публикация в bucket

- HTML ticket page и PDF должны генерироваться на Fly.io сразу после успешной регистрации. — Статус: `Зафиксировано`
- Вместе с HTML и PDF должен генерироваться статический `.ics`-файл для календарных интеграций. — Статус: `Зафиксировано`
- Генерация должна происходить из одного `ticket view-model`, чтобы HTML и PDF не расходились по данным. — Статус: `Зафиксировано`
- Для генерации не используется тяжёлый headless browser/Chromium pipeline. — Статус: `Зафиксировано`
- HTML-страница должна рендериться из лёгкого шаблона с подстановкой данных, PDF — из такого же view-model через лёгкий PDF-рендерер без HTML-to-PDF движка, а `.ics` — как текстовый календарный артефакт без тяжёлых зависимостей. — Статус: `Зафиксировано`
- Fly.io приложение получает bucket credentials только через secrets и само публикует HTML/PDF/ICS в S3-совместимое хранилище. — Статус: `Зафиксировано`
- В браузер никогда не отдаются bucket credentials или прямые права на upload. — Статус: `Зафиксировано`
- Если позже меняется билет или статус регистрации, ticket artifacts должны уметь регенерироваться сервером и перепубликовываться в bucket. — Статус: `Зафиксировано`
- Публикация из Fly.io разрешается только в выделенные префиксы bucket, например `tickets/` и `exports/`, без записи в корень сайта и без удаления чужих объектов. — Статус: `Зафиксировано`
- Если bucket общий с публичным сайтом, для Fly.io должны использоваться отдельные credentials с максимально узкой политикой доступа только к разрешённым префиксам. — Статус: `Зафиксировано`
- На production Fly.io не должно быть операций массового `list/delete/sync` по bucket root. Только адресная запись в собственные ключи. — Статус: `Зафиксировано`
- Рекомендуемый namespace ticket artifacts: `tickets/<public_hash>/index.html`, `tickets/<public_hash>/ticket.pdf`, `tickets/<public_hash>/event.ics`. — Статус: `Зафиксировано`
- Для Fly.io используется отдельный service key с правами только на `tickets/*` и `exports/*`; более широкие bucket credentials для приложения не допускаются. — Статус: `Зафиксировано`

## 13. Шифрование персональных данных и blind indexes

- В БД персональные данные хранятся только в зашифрованном виде. — Статус: `Зафиксировано`
- Для шифрования используется гибридная схема: per-record случайный симметричный ключ шифрует PII blob, а сам этот ключ заворачивается открытым ключом. — Статус: `Зафиксировано`
- Алгоритм шифрования payload: `AES-256-GCM`. — Статус: `Зафиксировано`
- Алгоритм шифрования data key открытым ключом: `RSA-OAEP-256`. — Статус: `Зафиксировано`
- Закрытый ключ для расшифровки хранится только в secrets Fly.io. — Статус: `Зафиксировано`
- Для MVP допустимо, что приложение физически имеет доступ к закрытому ключу, но дешифрование разрешено только в export/report-потоке и не используется для обычного browse/registration UI. — Статус: `Зафиксировано`
- Для проверки дублей и поиска по ФИО используются non-reversible blind indexes/fingerprints, а не plaintext-колонки. — Статус: `Зафиксировано`
- Fingerprints строятся как `HMAC-SHA256` от нормализованных значений и отдельного server-side secret, чтобы их нельзя было использовать как простой словарь plaintext-значений. — Статус: `Зафиксировано`
- Ticket page и PDF не должны требовать расшифровки из БД на каждый просмотр: их нужно генерировать на момент регистрации из входного payload и event data, а затем публиковать как статические артефакты. — Статус: `Зафиксировано`

## 14. API между публичным сайтом и Fly.io

- Публичный сайт на Яндекс-хостинге и Fly.io backend взаимодействуют только по отдельному HTTPS API. — Статус: `Зафиксировано`
- Для day-1 MVP допустим production API на штатном `.fly.dev` hostname с HTTPS; custom domain на Fly.io может быть добавлен позже без смены API-контракта. — Статус: `Зафиксировано`
- API должен быть versioned, например `/api/v1/register`, `/api/v1/ticket/<public_hash>/status`. — Статус: `Зафиксировано`
- На API нужны строгий origin allowlist, CORS только для production/staging origins, rate limit, request size limit и idempotency token для защиты от двойной отправки формы. — Статус: `Зафиксировано`
- Bucket upload, Telegram webhook и operator export работают только server-to-server и не зависят от браузерных секретов. — Статус: `Зафиксировано`
- Публичный ticket URL должен открываться на основном домене сайта, а не на внутреннем Fly host, даже если генерация идёт на Fly. — Статус: `Зафиксировано`
- Публичная форма сайта отправляет данные напрямую на Fly API по HTTPS; Telegram-команды управляют только администрированием и не участвуют в пользовательском submit-потоке. — Статус: `Зафиксировано`

## 15. Лёгкая защита API и ограничение нагрузки

- Все request bodies на публичном API должны быть очень маленькими; рекомендуемый лимит для registration POST — не более `4 KB`. — Статус: `Зафиксировано`
- Публичный API не принимает file uploads и multipart payloads. — Статус: `Зафиксировано`
- На Fly.io инстансе с `1024 MB RAM` все операции регистрации должны оставаться CPU- и memory-light. — Статус: `Зафиксировано`
- Для registration endpoint нужен лёгкий in-memory rate limit по IP и отдельный rate limit по event slug. — Статус: `Зафиксировано`
- Нужен глобальный concurrency cap на тяжёлые post-registration операции, например генерацию PDF и upload в bucket, чтобы не раздувать память при всплеске регистраций. — Статус: `Зафиксировано`
- Регистрация не должна тянуть за собой серверную обработку изображений, HTML-to-PDF рендер через браузер, ZIP-архивацию и другие тяжёлые задачи в синхронном пути. — Статус: `Зафиксировано`
- На уровне HTTP-сервера должны быть короткие timeouts для headers/body и ограничение keep-alive, чтобы slowloris-подобные запросы не съедали соединения. — Статус: `Зафиксировано`
- Все ответы API должны быть компактными JSON-объектами без тяжёлых вложений и без возврата персональных данных сверх необходимого минимума. — Статус: `Зафиксировано`
- Registration API должен быть stateless и не должен держать долгоживущие in-memory сессии пользователя между шагами формы. — Статус: `Зафиксировано`
- Публичный registration API по своей природе остаётся публичным; без CAPTCHA или доверенного промежуточного backend нельзя криптографически гарантировать, что запрос пришёл только “с сайта”. — Статус: `Зафиксировано`
- `CORS`, `Origin` и `Referer` checks рассматриваются как defence-in-depth, а не как полноценная аутентификация клиента. — Статус: `Зафиксировано`

### Рекомендуемые стартовые лимиты для Fly.io `1024 MB`

- `POST /api/v1/register`: burst `3` запроса за `30` секунд на IP и long-window `20` попыток за `15` минут на IP. — Статус: `Зафиксировано`
- Дополнительный per-event limit: не больше `2` одновременных registration POST от одного IP на одно событие. — Статус: `Зафиксировано`
- Concurrency cap на генерацию PDF и upload в bucket: `1-2` одновременные задачи на инстанс. — Статус: `Зафиксировано`
- `headersTimeout`: около `5s`, `bodyTimeout/requestTimeout`: около `10-15s`, `keepAliveTimeout`: около `5s`. — Статус: `Зафиксировано`
- Любые значения выше этих стартовых лимитов должны повышаться только после реальных замеров на staging. — Статус: `Зафиксировано`

- У формы должен быть не только финальный текст ошибки, но и видимый loading-state, чтобы пользователь никогда не оставался в неясности после клика по submit. — Статус: `Зафиксировано`

## 17. Базовая безопасность MVP

- Все публичные запросы идут только по HTTPS. — Статус: `Зафиксировано`
- Секреты бота и webhook-хедеры хранятся только в runtime secrets, не в клиентском коде. — Статус: `Зафиксировано`
- Telegram webhook должен проверяться по `secret_token` header. — Статус: `Зафиксировано`
- На регистрационный endpoint нужны как минимум request size limit, basic rate limit по IP, honeypot и журналирование ошибок. — Статус: `Зафиксировано`
- Пользовательский ввод должен безопасно экранироваться в UI, логах, XLSX и Telegram-сообщениях. — Статус: `Зафиксировано`
- Согласие на обработку персональных данных должно акцептоваться через отдельный checkbox, а не подразумеваться самим фактом submit. — Статус: `Зафиксировано`
- На форме должны быть доступны ссылки на актуальную редакцию согласия и политику конфиденциальности/обработки персональных данных. — Статус: `Зафиксировано`
- Рабочая редакция согласия должна быть основана на официальном PDF Российского общества «Знание» и адаптирована под фестиваль с формулировкой про участие Калининградского филиала Российского общества «Знание». — Статус: `Зафиксировано`
- До запуска финальную редакцию текста согласия всё равно должен утвердить владелец проекта или юрист. — Статус: `В доработке`
- Аварийный export endpoint не должен передавать секрет через query string; авторизация должна идти через заголовок `Authorization`. — Статус: `Зафиксировано`

## 18. Сохранность данных и ежедневные backup-процедуры

- Перезапуск контейнера или redeploy приложения не должен приводить к потере данных; SQLite-файл живёт на persistent Fly Volume. — Статус: `Зафиксировано`
- Production-приложение не должно молча создавать новую пустую базу поверх неожиданно отсутствующего volume после первичного запуска; в таком случае приложение должно падать в health checks и требовать явного вмешательства. — Статус: `Зафиксировано`
- Ежедневный backup должен запускаться в окно минимальной нагрузки по времени `Europe/Kaliningrad`. Для day-1 MVP фиксируется время `04:30`. — Статус: `Зафиксировано`
- В ежедневный пакет входят как минимум сводный XLSX по всем билетам и backup SQLite / SQL dump для суперадмина в Telegram. — Статус: `Зафиксировано`
- Метка последнего успешного backup-запуска хранится в БД и используется для мониторинга, чтобы видеть пропуски расписания. — Статус: `Зафиксировано`

## 19. Что сознательно не включаем в MVP

- Multi-region и multi-writer схему. — Статус: `Зафиксировано`
- Полноценную web-админку. — Статус: `Зафиксировано`
- Redis/BullMQ/RabbitMQ. — Статус: `Зафиксировано`
- Waitlist и автоматическое освобождение мест с переприглашением. — Статус: `Зафиксировано`
- Личный кабинет пользователя. — Статус: `Зафиксировано`
- Сложную email-автоматизацию сверх базового ticket flow, пока это отдельно не подтверждено. — Статус: `Зафиксировано`
- QR-коды и check-in контур. — Статус: `Зафиксировано`
- Команду ручной отмены регистрации суперадмином в day-1 MVP. — Статус: `Зафиксировано`

## 20. Тестовые режимы, очистка тестовых данных и prod-first запуск

- Для day-1 MVP допускается production-first запуск, но это не отменяет разделения между локальным destructive-тестированием и публичными прогонами на живом приложении. — Статус: `Зафиксировано`
- Полный конкурентный, нагрузочный и destructive E2E-контур должен выполняться локально или на закрытом staging-контуре с отдельной SQLite-базой. — Статус: `Зафиксировано`
- На production допустимы smoke/E2E-прогоны только по выделенным test events, которые не используются для реальных посетителей. — Статус: `Зафиксировано`
- Автоматические тесты на production не должны использовать реальные visitor events и не должны загрязнять реальные отчёты для действующих регистраций. — Статус: `Зафиксировано`
- Каждому автоматическому прогону присваивается `test_run_id`, который сохраняется в регистрациях этого прогона и используется для очистки тестовых данных. — Статус: `Зафиксировано`
- Для тестовых прогонов используются отдельные тестовые значения ФИО/email/телефона с предсказуемым префиксом или plus-alias схемой, чтобы их можно было однозначно отличить в админских выборках. — Статус: `Зафиксировано`
- Для очистки тестовых данных нужен superadmin-only maintenance сценарий `cleanup_test_run`, который удаляет тестовые регистрации, тестовые билеты, связанные bucket artifacts и пересчитывает `seats_taken`. — Статус: `Зафиксировано`
- Очистка тестовых данных не должна быть доступна через публичный registration API. — Статус: `Зафиксировано`

## 21. Подводные камни, которые нужно учитывать заранее

- Single-machine Fly.io схема ускоряет запуск, но не защищает от host failure и даёт короткий downtime на deploy. — Статус: `Зафиксировано`
- Telegram webhook требует публичный HTTPS endpoint; полноценный E2E без Fly staging не собрать только на localhost. — Статус: `Зафиксировано`
- Переключение бизнес-правила дублей с per-event на global в будущем потребует миграции blind indexes, unique constraints и UX-сообщений, поэтому текущее per-event правило нельзя менять “по ходу дела” без отдельной ревизии требований. — Статус: `Зафиксировано`
- XLSX и SQL dump не должны запускаться внутри транзакции записи и в пиковой синхронной ветке регистрации. — Статус: `Зафиксировано`
- Общий bucket для сайта и ticket artifacts требует особенно жёсткой изоляции по префиксам и правам, иначе ошибка deploy/upload может затронуть публичный сайт. — Статус: `Зафиксировано`
- Требование “в БД только шифротекст” конфликтует с поиском, дублями и ticket UX, если не заложить blind indexes и генерацию ticket artifacts на момент регистрации. — Статус: `Зафиксировано`
- Короткий `short_ticket_id` удобен человеку, но не должен использоваться как публичный секрет или основной URL-ключ; для доступа к билету всегда используется только длинный `public_hash`. — Статус: `Зафиксировано`
- Календарные ссылки нельзя собирать на клиенте из произвольных данных пользователя; все calendar URLs и `.ics`-артефакт должны генерироваться сервером из уже нормализованных event data. — Статус: `Зафиксировано`
- Если registration-UI строится только на локальном клиентском состоянии, кнопки `Регистрация`, `Скоро откроется` и `Закрыта` начнут врать при смене статуса из Telegram; источник истины должен быть только серверный event-state. — Статус: `Зафиксировано`
- Если не сохранять exact version/hash текста согласия, потом будет невозможно доказать, какую именно редакцию пользователь акцептовал в день регистрации. — Статус: `Зафиксировано`
- При регенерации ticket HTML/PDF в общем bucket есть риск stale-кэша на стороне публичной раздачи; нужны понятные cache-control правила и предсказуемая стратегия overwrite. — Статус: `Зафиксировано`
- Слишком агрессивный IP rate-limit может ложно резать мобильных пользователей за NAT или посетителей одной корпоративной сети; лимиты нужно проверять на staging реальным трафиком. — Статус: `Зафиксировано`
- Disposable-email список может устаревать и давать ложные блокировки; нужен контролируемый update-механизм и способ ручной проверки спорных доменов. — Статус: `Зафиксировано`
- Успешный commit регистрации и неуспешная публикация ticket artifacts в bucket создадут “полузарегистрированное” состояние; нужен repair/retry контур и отдельный мониторинг таких сбоев. — Статус: `Зафиксировано`
- Telegram может не принять слишком крупный XLSX или backup-файл; для таких случаев нужен fallback на публикацию export-артефакта в `exports/` и отправку суперадмину защищённой ссылки вместо вложения. — Статус: `Зафиксировано`
- Если ссылка на публичную редакцию согласия или PDF ведёт в 404/устаревшую версию, юридическая связка consent-checkbox и фактического текста ломается; ссылки и versioning consent-документа нужно проверять автоматически. — Статус: `Зафиксировано`

## 22. Остаток перед реализацией

- Документация для day-1 MVP считается финализированной и готовой к реализации. — Статус: `Зафиксировано`
- Ручная юридическая правка текста согласия после согласования с юристом входит в согласованный предзапусковый scope и не требует пересмотра архитектуры, если не меняются сами категории данных, цели обработки и роли участников процесса. — Статус: `Зафиксировано`
- Любая ручная правка текста согласия обязана сопровождаться новым `consent_version`, обновлением `consent_text_hash`, регенерацией PDF, проверкой ссылок из checkbox-блока и повторной smoke-проверкой формы регистрации. — Статус: `Зафиксировано`

## 23. Минимальные secrets для Fly.io

Ниже — минимальный набор именно secrets. Это не полный список env/config приложения.

### Обязательные Fly secrets

- `TELEGRAM_BOT_TOKEN` — токен Telegram-бота.
- `TELEGRAM_WEBHOOK_SECRET` — секрет для проверки Telegram webhook.
- `EMERGENCY_EXPORT_TOKEN` — bearer token аварийного read-only export endpoint.
- `PII_PRIVATE_KEY_PEM_B64` — закрытый ключ для расшифровки PII, сохранённый в base64.
- `PII_PUBLIC_KEY_PEM_B64` — открытый ключ для шифрования PII, тоже удобно хранить в base64 на том же Fly-приложении.
- `PII_FINGERPRINT_SECRET` — HMAC-secret для blind indexes по ФИО, email и телефону.
- `S3_ACCESS_KEY_ID` — access key для записи только в `tickets/*` и `exports/*`.
- `S3_SECRET_ACCESS_KEY` — secret key для того же bucket user/service account.
- `TELEGRAM_SESSION_S22` не относится к production secrets Fly.io и не должен туда попадать; он нужен только локальному E2E-контуру с Telethon. — Статус: `Зафиксировано`

### Обычно тоже лучше хранить как secrets

- `BOT_WEBAPP_AUTH_SECRET` — если позже появится подписанный internal bot/web auth exchange.
- `BACKUP_ARCHIVE_SECRET` — если backup/export файлы будут дополнительно шифроваться перед отправкой или публикацией.

### Нужны приложению, но не обязаны быть secrets

- `APP_BASE_URL`
- `PUBLIC_SITE_BASE_URL`
- `S3_BUCKET`
- `S3_ENDPOINT`
- `S3_REGION`
- `TICKETS_PREFIX`
- `EXPORTS_PREFIX`
- `TZ=Europe/Kaliningrad`
